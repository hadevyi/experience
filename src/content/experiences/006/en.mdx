  overview: |
    O to I was a health management web service for healthier dieting and lifestyle management. Users could enter height, weight, target weight, constitution, and diet information, then check BMI, calculate diet calories, and view different Diet content depending on their login status.

    The goal was not only to show static information, but also to let users enter their own status, follow diet and exercise recommendation flows, and leave records through community and customer support features.
  background: |
    At the time, wellness and healthy living were becoming common topics of interest, and the class project allowed each team to choose its own theme. Through team discussions, we chose to build a health management service that recommended diet and exercise information based on each user's constitution and status, rather than focusing only on weight loss.

    Because the team consisted of first-year students, defining the implementation scope and dividing work into manageable pieces became an important part of the project.
  scope: |
    According to the wiki, the main scope included About O to I, Diet, BMI measurement, member-only Diet, Community, and Customer Service. The member-only Diet flow aimed to calculate calories by menu and allow new menu-calorie registration, while Customer Service included member inquiries and admin replies.

    The codebase includes signup and login, constitution calculation, BMI calculation, Food_db-based diet calorie totals, guestbook and free board features, customer inquiries and admin replies, an admin member list, and member deletion. However, not every planned feature reached the same level of finish. The admin page and inquiry flow existed in code, but remained areas where the polish and completion level fell short of the original plan.

  implementation: |
    The project used an ASP.NET Web Forms structure with .aspx pages and code-behind files. Master Pages were used to share layout across menus. It was also a concept covered in class, so I applied it to keep the overall layout consistent across the service.

    Member and non-member screens were separated so visitors could try some features before signing up, while saved or extended functionality could lead naturally into registration. The code also includes separate flows based on login status and Forms Authentication-based login handling.

    Based on the code, the data layer used a local SQL Server SQLEXPRESS database named sqlexpress2. During signup, user data was inserted into user_db, and questionnaire answers were used to calculate and store each user's constitution value.

    In the Diet area, BMI was calculated from height and weight, and selected menu calories were summed from Food_db. The goal was to make changing user numbers, such as BMI and diet calories, visible in the service flow.
  challenges: |
    The hardest part was database design. As implementation progressed, new data needs kept appearing, and we had to repeatedly revisit tables and relationships.

    Because this was my first time handling different types of data together, including user information, constitution, diet, boards, and customer support, the database structure changed often. When the design became unclear, I used the data mind map approach we were learning in class to reorganize the data and relationships, then discussed the table and feature scope again with the team.
  outcome: |
    The project received the best evaluation in the class, and that result gave me confidence during my first major project.

    At the same time, we were only able to implement about 80% of what we had planned. The admin page and customer support flow had basic code paths, but they remained areas that did not reach the level of completion I had hoped for. Looking back, this was my first web project and my first experience leading development. It became a starting point for understanding that schedule coordination and scope management matter just as much as the final result.
