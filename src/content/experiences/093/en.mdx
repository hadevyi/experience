title: Git Toy Project Mentoring for an Industry-Academic Project
organization: Inje University SW-Centered University Program
summary: Mentored an industry-academic project team as they built self-directed toy projects and practiced Git-based collaboration. The mentoring combined regular sessions with GitHub feedback to address questions that came up during hands-on work.
oneLine: Mentoring that helped a toy project team learn Git collaboration workflows
highlights:
  - "Led team mentoring tailored to each student’s Git understanding"
  - "Connected basic commands, PRs, and Issues through practice and feedback"
  - "Provided ongoing written feedback through GitHub Issue comments and PR reviews"
teachingMentoring:
  audience: |
    This mentoring was for one team participating in an industry-academic project.
    The team had four students, including one international student, and most participants were in their second or third year.
  reason: |
    I joined to help the team use Git not only as a storage tool, but as a way to document progress and share changes.
    The goal was to help them build collaboration habits while working on their project.
  content: |
    I covered Git usage in the context of each student's self-directed toy project.

    - Basic Git commands
    - Creating and managing GitHub Issues
    - Opening Pull Requests and reviewing changes
    - Feedback on questions and errors that came up during development
  approach: |
    I ran four regular mentoring sessions, each about one hour long.
    The sessions combined concept explanation, hands-on practice, and real-time feedback while students tried the workflow themselves.

    Outside the regular sessions, I answered questions and gave written feedback through GitHub Issue comments and Pull Request reviews.
    When students got stuck with code or Git usage, I checked the situation and guided them on how to organize the next step.
  response: |
    After the mentoring, one junior student started organizing their Git usage and study notes more actively.
    It was meaningful to see the student move beyond copying commands and begin documenting the workflow they had practiced.
  learning: |
    I confirmed that Git is difficult to connect to real project work when it is taught only as a list of commands.
    Concepts such as `remote`, which require understanding the relationship between local and remote repositories, needed hands-on connection and verification.
    Collaboration features became easier to understand when students applied them directly in a project context.
