  overview: |
    Gratz was a web service designed to make personal anniversaries feel more special.
    Users could choose an anniversary theme, share a link, and receive celebration messages and gifts from friends.

    The service was structured so users could check messages while waiting for the date and later save the anniversary as a memory.
  background: |
    The project was built during DND 6th cohort with a goal of fast implementation and real service adoption.
    It aimed to deliver a special experience for receiving support and celebration even when people could not celebrate in person.

    The initial idea was decided quickly, but the planning and design had to be revised multiple times to avoid concerns around copying a similar service that was an issue at the time.
  scope: |
    The project included a web service, backend APIs, database design, common code and metadata, the message domain, image storage, and deployment infrastructure.
  contribution: |
    I participated as a Backend / Server Developer.
    The backend team jointly designed the database and API documentation, aligning request and response structures with frontend developers.

    My individual responsibilities included AWS EC2 instance setup, S3, domain setup, Backend/Frontend Jenkins CI/CD, common code and metadata design and implementation, JPA-based database implementation, and Message API CRUD.
  roleDetails:
    - Participated as Backend / Server Developer
    - Shared database design and API docs design
    - Built AWS EC2, S3, and domain setup
    - Built Backend/Frontend Jenkins CI/CD
    - Designed and implemented common code and metadata
    - Implemented the database layer with JPA
    - Implemented Message API CRUD
  implementation: |
    The backend used Java 11, Spring Boot, MySQL, JPA, and Spring Security.
    It included Kakao social login and handled domains such as messages, events, users, scraps, common codes, and gift themes.

    The infrastructure was built on AWS EC2 and S3, with Nginx and Jenkins used for frontend and backend build/deployment flows.
    The project record also shows that the EC2 instance type was adjusted after Jenkins CI/CD caused CPU overload.
  challenges: |
    The biggest regret was that repeated planning and design changes shortened the actual development time.
    To differentiate the service from similar products, we had to revise user flows, feature priorities, and implementation scope.

    Technically, the work went beyond API development because AWS, domain setup, Jenkins CI/CD, and frontend/backend deployment flows had to be handled together.
  outcome: |
    The project was completed as DND 6th cohort Team 9. It did not continue into full operation, but we proceeded to beta testing.
    During testing, we received feedback that the service felt light and fun, and beta testers actually used the service flow.

    Through this experience, I learned that opening a service requires more than promotion and usability. It also requires considering external contexts such as differentiation from similar services and copycat concerns.
    I also experienced API documentation, deployment automation, and frontend-backend collaboration in one project cycle.
